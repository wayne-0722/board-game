# 桌遊硬體配套即時答題系統

## 專案定位

本專案是為實體桌遊活動開發的即時答題與計分系統。實體桌遊硬體負責現場互動與遊戲體驗，Web 系統負責多人加入、題目同步、答題判定、搶答、分數計算與回合推進。

作品集呈現重點不是「做了一個遊戲網頁」，而是「將實體桌遊流程數位化，並解決多人即時同步與手機操作的問題」。

## 開發動機

實體桌遊在多人活動中常見幾個問題：

- 人數多時，主持人手動判定答題順序與分數容易出錯。
- 多位玩家同時參與時，題目、回合與搶答狀態需要即時同步。
- 玩家主要使用手機操作，介面必須適合現場快速點擊。
- 活動不需要長期保存資料，但需要單場流程穩定。

因此本系統採用手機 Web + Socket.IO 的方式，讓玩家不需要安裝 App，只要透過網址加入房間即可參與。

## 系統架構

```txt
玩家手機瀏覽器
    |
    | HTTPS / Socket.IO
    v
Render Web Service
    |
    | Node.js custom server
    v
Next.js + Socket.IO Server
    |
    | Session state
    v
In-memory store / optional Redis
```

前端使用 Next.js、React、TypeScript 與 Tailwind CSS。後端使用 Node.js 自訂 server，整合 Socket.IO 進行即時同步。狀態管理分成前端 Zustand store 與後端 session store。

## 技術選型

- `Next.js 14`：負責頁面、路由與前端建置。
- `React`：建立玩家操作畫面。
- `TypeScript`：降低遊戲流程與事件資料結構錯誤。
- `Tailwind CSS`：快速調整手機版 UI。
- `Zustand`：管理前端 session state 與 socket connection state。
- `Socket.IO`：處理房間同步、玩家事件與伺服器推送。
- `Node.js custom server`：讓 Next.js 與 Socket.IO 共用同一個服務。
- `Render`：部署成可對外連線的 Node Web Service。

## 核心流程

1. 玩家使用手機進入網站。
2. 系統顯示活動入口圖，點擊入口後進入加入房間畫面。
3. 玩家輸入房號並建立 Socket.IO 連線。
4. 伺服器將玩家加入對應 session room。
5. 主流程進入 lobby、答題、搶答、反思與結算階段。
6. 每次答題、搶答或回合推進後，伺服器重新廣播最新 session state。
7. 同一房間的玩家畫面保持同步。

## 即時同步設計

本專案原本若只使用 HTTP API，會遇到玩家端狀態更新延遲與輪詢成本問題。因此改用 Socket.IO，讓伺服器在狀態變更後主動推送給房間內玩家。

Socket.IO 事件主要負責：

- 建立連線與重連。
- 加入指定房號。
- 開始或推進回合。
- 送出答案。
- 開啟與處理搶答。
- 更新分數。
- 反思階段提交與結算。

這種設計適合活動現場，因為玩家手機只要維持連線，就能即時收到狀態更新，不需要一直手動重新整理。

## 遊戲狀態管理

後端以 session 為單位管理遊戲狀態，包含：

- 房號與玩家清單。
- 目前階段。
- 目前題目與作答者。
- 答題結果。
- 搶答視窗是否開啟。
- 分數與排名。
- 反思階段狀態。

目前預設使用記憶體儲存，適合單場約 100 人、不需要長期留存的活動。系統也保留 `REDIS_URL` 支援，未來若需要重啟後保留房間資料，或需要多台 instance 擴展，可以改接 Redis。

## 手機操作設計

主要使用者是活動現場玩家，因此 UI 以手機優先設計。首頁在手機版顯示活動入口圖，並把圖中主要入口區域做成可點擊按鈕；電腦版則略過入口圖，直接顯示測試用加入畫面。

搶答按鈕原本如果放在頁面底部，手機玩家在現場操作時不容易快速點擊。因此改成浮動視窗，讓搶答行為更接近活動中的即時反應。

## 流程防呆

系統在後端加入流程保護，避免只靠前端按鈕控制造成規則被繞過。例如錯答後如果搶答視窗已開啟，原答題者不能直接推進下一回合，必須等搶答流程結束。這類邏輯放在後端驗證，確保不同玩家手機畫面或延遲狀況不會破壞規則。

## 部署方式

本專案部署在 Render Web Service，原因是系統需要維持 Socket.IO 連線，不適合只做靜態網站部署。

部署設定：

- Build command：`npm ci && npm run build`
- Start command：`npm start`
- Node version：20
- Service type：Node Web Service
- Public URL：`https://board-game-oclb.onrender.com`

單場活動使用一台 Render instance 即可。如果未來要開多台 instance，需要加入 Redis 與 Socket.IO adapter，避免同一房間的玩家被分配到不同 instance 後無法共享狀態。

## 成果

系統已可支援玩家透過手機加入同一場桌遊活動，並即時同步題目、答題結果、搶答狀態與分數。對作品集而言，本專案展示了即時通訊、多人狀態同步、手機優先 UI、流程防呆與公網部署能力。

## 可延伸方向

- 加入 Redis + Socket.IO adapter，支援多台 instance 擴展。
- 加入活動後台，讓主持人可視覺化控制題目與回合。
- 加入活動結果匯出，例如 CSV 或簡易排行榜頁面。
- 加入硬體事件串接，讓實體桌遊裝置直接觸發 Web 系統事件。
